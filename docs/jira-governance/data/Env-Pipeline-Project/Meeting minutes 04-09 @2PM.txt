1. Context & Objective (5 min)

“Thanks everyone for joining.
The goal of this meeting is alignment — not just discussion.

We’ve been talking about environment pipelines, UI smoke tests, and deployment confidence, but some key decisions are still open-ended. As we scale MFEs and microservices, we need a shared understanding of:

What we test in UI vs API
What qualifies as a smoke test
What ‘pass’ means for deployment confidence
And what risks QA and Tech Leads are willing to accept

Today, we want to walk out with agreement — or at least clear ownership — on these points so we can finalize this approach and operationalize it.”

2. UI vs API Coverage & Scope Definition (7 min)

“One major open question is why a UI test exists when API tests already validate responses.

From QA’s perspective, UI tests are inherently integration-level validations — they confirm:

Correct wiring between frontend, backend, auth, and data
Schema usage in real user flows
Critical regressions that API tests can’t expose

That said, UI tests are more expensive and less stable, so we can’t treat them the same as API tests.

What we need alignment on today:

What criteria make a UI test ‘smoke worthy’?
Is one UI scenario per URN/schema sufficient — or not?
Who decides when UI coverage is ‘good enough’?

Today’s proposal is that QA Leads define coverage, in close collaboration with Tech Leads, with explicit risk acknowledgment — not implicit assumptions.”

3. Pass/Fail Criteria, Risk & Stability Handling (7 min)
“This is the most critical section.

UI tests introduce instability — flaky elements, timing issues, grid variability.
But our pipelines expect deterministic pass/fail outcomes.

So we need to define:

What happens when UI fails for non-functional reasons?
Do we treat flaky failures differently than real regressions?
Do all UI smoke tests block deployments?
What is the escalation or override path?

Today, we don’t have a defined process for:

Test instability
False positives
Temporary disablement vs long-term fixes

If a UI smoke suite passes, we should be confident enough to deploy.
If not — we need clarity on what breaks the pipeline and why.

This requires explicit risk decisions from QA, not assumptions baked into pipelines.”

4. Execution Constraints & Grid Capacity (6 min)

“We also need to ground this in reality.

We’ve said the smoke suite must complete in 3–5 minutes.
That immediately limits:

Number of UI tests
Parallelism
Grid usage

With current grid capacity, we’re realistically talking:


~10–15 UI tests in parallel execution
Possibly up to ~30 only under ideal grid availability

We need alignment on:

Maximum acceptable execution time
Parallel load expectations
Whether grid expansion is needed
How cost efficiency factors into capacity decisions

This is not just a QA concern — it directly impacts release velocity.”

5. Decisions, Ownership & Next Steps (5 min)

“To close, here’s what we want coming out of this meeting:

✅ Agreement or clear ownership on:

UI smoke test inclusion criteria
Pass/fail definition and risk tolerance
Handling instability and failures
Execution limits and grid expectations

✅ Clear action items with owners:

Documenting the agreed-on criteria
Mapping API tests to UI smoke cases
Updating pipeline logic if needed

The intent is to finalize this model once, document it, and reuse it across teams — not re-debate it every release.”
